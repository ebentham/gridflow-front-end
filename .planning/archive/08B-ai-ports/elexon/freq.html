Overview

The GB transmission system must maintain frequency within 49.5–50.5 Hz at all times. FREQ exposes the telemetry stream as 1-minute aggregates, giving a continuous record of how well generation and demand are matched in real time.

In gridflow the dataset is ingested by FreqConnector, validated against ElexonFrequency (ge=49.0, le=51.0 Hz bounds), and written as Parquet partitioned by year/month. The silver layer holds a single numeric column per timestamp — lean enough to scan years of history in seconds.

Typical uses: input feature for frequency-response cost models, regressor alongside imbalance price, anomaly detection for frequency-keeping obligation assessment.

Caveats

01
V2-FIX-01 — param-name mismatch (resolved 2026-05-09)

The connector previously sent incorrect query parameter names (from/to instead of measurementDateTimeFrom/measurementDateTimeTo). The API silently ignored them and returned the latest 5761 samples regardless of the requested window. Fixed in gridflow commit fix(V2-A). Pre-fix bronze files hold the wrong window — re-ingest historical FREQ to get correctly-windowed data.

02
High-frequency volume

~5760 samples per 3-hour window. Bronze files can grow large over long historical runs. Partition by hour or use a streaming silver pipeline when querying multi-week ranges.
