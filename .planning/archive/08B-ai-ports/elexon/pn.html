Overview

Physical Notifications are the MW level each BM unit declares it intends to generate (or consume) in a settlement period. They are submitted before gate closure (~1 hour ahead) and form the pre-dispatch baseline for the period. Every active BM unit submits a PN, producing approximately 2,500 rows per period across all 50 periods per day.

The silver schema carries level_from and level_to — the MW level at the start and end of the period respectively — which together describe the ramp profile within the half-hour. The primary key is the three-field composite (settlement_date, settlement_period, bm_unit_id).

PN is the unit-level baseline; BOAL acceptances are the deviations from PN that NESO agreed. Joining PN to BOAL on bm_unit_id reconstructs the accepted vs notified dispatch picture for any unit.

API access

GET https://data.elexon.co.uk/bmrs/api/v1/datasets/PN
  ?settlementDate=2026-05-06
  &settlementPeriod=24
  &format=json
SELECT settlement_date, settlement_period, bm_unit_id,
       level_from, level_to
FROM read_parquet('data/silver/elexon/pn/**/*.parquet')
WHERE settlement_date = '2026-05-06'
  AND settlement_period = 24
ORDER BY bm_unit_id;
import polars as pl

df = pl.read_parquet("data/silver/elexon/pn/**/*.parquet")
print(df.filter(
    (pl.col("settlement_date") == pl.date(2026, 5, 6)) &
    (pl.col("settlement_period") == 24)
).sort("bm_unit_id"))

Caveats

01
Per-period fetch — high row count and volume

The connector iterates periods 1–50 for each settlement date, fetching ~2,500 rows per period (one per active BM unit). A single day of PN yields approximately 125,000 rows. Partition by settlement_date and filter aggressively in analytical queries.

02
Stop-on-empty optimisation — DST short days

The connector stops iterating when a period returns empty data, which handles DST spring-forward days (46 periods) naturally. Autumn-clock-change days have 50 periods.
