## Manual QA Approach

This is the entire QA story for the repo today. It has three parts:

**1. Local serve with `gridflow-serve`** (`src/gridflow_front_end/serve.py`)
- `uv run gridflow-serve` starts a stdlib `http.server.HTTPServer` rooted at `site/hifi/` and opens the default browser at `http://localhost:8765/` after a 0.5 s delay (`serve.py:21-23`, `:38-48`, `:51-100`).
- The `_SilentHandler` subclass suppresses per-request access logs but still surfaces real errors to stderr (`serve.py:26-35`), so the console stays readable while the developer clicks through pages.
- Ctrl-C exits cleanly via the `KeyboardInterrupt` branch (`serve.py:98-100`).
- Port conflicts are caught (`OSError`) and reported with a remediation hint (`serve.py:84-88`).

**2. Visual review against `.codex-assessment-shots/`**
The `.codex-assessment-shots/` directory at the repo root holds reference PNG screenshots used for eyeballed regression checks:
- `.codex-assessment-shots/architecture.png` — the `/architecture.html` page.
- `.codex-assessment-shots/elexon.png` — the vendor page `/data-sources/elexon.html`.
- `.codex-assessment-shots/index.png` — the home `/index.html`.
- `.codex-assessment-shots/sources.png` — `/data-sources.html`.

These four screenshots represent the design baseline. After any visual change, manually compare the served page against the corresponding PNG. The `data-screen-label` attribute on every `<body>` (e.g. `data-screen-label="06 Dataset · agpt"` at `site/hifi/data-sources/elexon/agpt.html:49`) provides a stable label so screenshots are easy to match to pages. There is no scripted screenshot diff tool — comparison is visual.

The directory is git-ignored implicitly (not in `.gitignore` but currently untracked per `git status`); treat it as a local reference rather than a committed artefact.

**3. GitHub Pages deploy as integration check**
- `.github/workflows/deploy.yml` runs on every push to `main` and on manual dispatch. It uploads `site/hifi/` as a Pages artefact and deploys it. There is no test or lint step before deploy.
- The deployed URL is the only end-to-end "is this real" check. A broken page only fails when a human notices.

## Coverage Gaps

This is a documentation site with a tiny Python dev-server wrapper, so the absence of automated tests is partly defensible. The biggest unverified surfaces:

- **HTML well-formedness across dataset pages.** Stub-format dataset pages (`boal.html`, `freq.html`, `indo.html`, `system_prices.html`, and others) open `<nav class="sidebar">` but close with `</div>`, leaving the `nav` element unclosed in source. No validator catches this in CI. See `.planning/codebase/CONCERNS.md`.
- **Internal link integrity.** Pages cross-reference each other heavily (`data-root` prefixes, relative `href`s, sidebar sibling-dataset lists). A broken `href` is invisible until a user clicks it.
- **`data-page` contract.** A new page that forgets `data-page` / `data-root` on `<body>` silently loses its global nav and footer (`site/hifi/assets/site.js:5-7`, `:84-98`). No check enforces this contract.
- **JSON content vs HTML content drift.** `site/hifi/data/elexon.json` lists datasets and counts but is not loaded at runtime; figures like "28 datasets" / "49 datasets" are duplicated in HTML copy (`site/hifi/index.html:373`, `site/hifi/data-sources.html:25`). When the JSON changes, HTML must be hand-updated.
- **Chart placeholder rendering.** `data-chart="…"` placeholders are rendered by `site/hifi/assets/charts.js` only if a matching renderer exists in that file. A typo in `data-chart` silently leaves an empty div.
- **Python serve.py behaviour.** The `OSError` bind path (`serve.py:84-88`) and the `_SITE_DIR.is_dir()` guard (`serve.py:70-74`) are not exercised by any test. Both have user-facing error messages worth pinning.

## Suggested Minimum Viable Test Plan (not yet implemented)

If/when tests are added, the cheapest high-value additions would be:

- **HTML validation:** `htmlhint` or `html5validator` over `site/hifi/**/*.html` in CI, before the Pages deploy.
- **Internal link check:** `lychee` or `linkchecker` against the built site.
- **`data-page` / `data-root` lint:** a 20-line Python or Node script that walks `site/hifi/**/*.html` and asserts every `<body>` has both attributes and that `data-page` is in the allowed set (`home|architecture|sources|vendor|dataset|model|about`, mirroring `site/hifi/assets/site.js:84-93`).
- **`pytest` for `serve.py`:** unit tests for `_SilentHandler.log_message` (silent), `_SilentHandler.log_error` (writes to stderr), `main()` exit on missing `_SITE_DIR`, and `main()` exit on port-in-use. Use `pytest -x -q` per project preference.
- **Visual regression:** swap `.codex-assessment-shots/` for a Playwright-driven screenshot diff in CI if the eyeball-comparison loop becomes too slow.

None of this is required by the codebase today; documenting it here in case it becomes a future phase.
