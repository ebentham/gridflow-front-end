Layers

Bronze · Silver · Gold.

Each layer has a strict contract. Nothing skips a layer. Nothing mutates a layer from outside.

Bronze
Raw.
Immutable.

Raw API responses, written exactly as received. JSON, XML, or CSV — whatever the vendor sends. Nothing is parsed or validated here. Every write appends; nothing is ever overwritten or deleted.

INVARIANTS
append-only · no transforms
one file per API response
SHA-256 content hash in filename
EXAMPLE PATH
bronze/elexon/system_prices/
2026/04/29/
raw_20260429T140000Z_a1b2c3d4.json
raw_20260429T140000Z_a1b2c3d4.meta.json
Silver
Typed.
Deduplicated.

Bronze files are read, parsed, renamed to snake_case, cast to strict types, and deduplicated by primary key. One Parquet file per dataset per calendar date, written atomically via a temp-then-rename pattern.

INVARIANTS
typed schema · no raw strings
Hive-partitioned by year/month
atomic write via os.replace()
EXAMPLE PATH
silver/elexon/system_prices/
year=2026/month=04/
system_prices_20260429.parquet
Gold
Joined.
Enriched.

Cross-source joins and derived features. Temporal features (hour of day, settlement period flags), price spreads, weather-adjusted demand. DuckDB SQL views or Polars builders — whichever suits the join complexity.

INVARIANTS
reads only from silver
no external API calls
documented derivation for every column
EXAMPLE PATH
gold/system_marginal_price/
year=2026/
system_marginal_price_20260429.parquet

End-to-end trace

One dataset,
six steps.

Following system_prices — Elexon's half-hourly system buy/sell price — from API call to SQL query. Every dataset in the pipeline follows this exact same path.

DATASET
elexon · system_prices
WHAT IT IS

Half-hourly system buy (SBP) and sell (SSP) prices used to settle imbalance. The reference price for all balancing mechanism analysis.

01
CLI invocation
Command dispatched.

The CLI resolves the date range, imports the connector module (which auto-registers it), and opens an async context with a rate-limited httpx client.

gridflow ingest elexon system_prices --last 24h

# internally: resolves to start=2026-04-28T14:00Z, end=2026-04-29T14:00Z
# imports connectors/elexon/client.py → triggers register_connector()
# opens httpx.AsyncClient with Semaphore(rate_limit_per_second=2)
Each source is a registered module — no central wiring file
02
Connector · Bronze layer
API fetched asynchronously.

The connector pages through the API, returning an immutable RawResponse per page — raw bytes plus full provenance metadata. Rate limiting is enforced by an asyncio semaphore before every request.

# connectors/base.py
@dataclass(frozen=True)
class RawResponse:
    body:           bytes        # raw API payload, untouched
    content_type:   str          # application/json | text/xml | text/csv
    source:         str          # "elexon"
    dataset:        str          # "system_prices"
    fetched_at:     datetime     # UTC timestamp of fetch
    request_url:    str          # full URL including query params
    http_status:    int          # 200 / 429 / 5xx
    page:           int          # pagination cursor
    data_date:      date | None  # calendar date the data refers to
Frozen dataclass — immutable after creation
03
Bronze write
Raw bytes land on disk.

BronzeWriter.write() computes a SHA-256 hash of the response body and uses the first 8 hex chars in the filename — so identical fetches are content-addressable and duplicates are trivially detectable. A .meta.json sidecar records the full request provenance alongside every file.

# bronze/writer.py — filename construction
body_hash = hashlib.sha256(response.body).hexdigest()[:8]
ts        = response.fetched_at.strftime("%Y%m%dT%H%M%SZ")
filename  = f"raw_{ts}_{body_hash}.json"

# directory partitioned by data date, not ingest date
dir_path = bronze / source / dataset / f"{year}/{month:02d}/{day:02d}"

# sidecar carries full provenance
meta = { "request_url": ..., "body_sha256": ...,
         "body_size_bytes": ..., "http_status": ... }
Content-hashed filename — idempotent re-fetch is a no-op
04
Silver transform
Typed, deduped, settled.

The transformer reads all bronze files for the target date, renames camelCase API fields to snake_case, casts every column to a strict Polars type, derives a UTC timestamp from the settlement date + period, then resolves run-type precedence. Elexon revises half-hourly prices through seven run types (II→SF→R1→R2→R3→RF→DF); only the highest-precedence run per settlement period is kept. The output is written atomically via a temp file and os.replace(). 

# silver/elexon/system_prices.py
RUN_PRECEDENCE = {"II": 1, "SF": 2, "R1": 3,
                  "R2": 4, "R3": 5, "RF": 6, "DF": 7}

def _resolve_runs(self, df: pl.DataFrame) -> pl.DataFrame:
    # keep only the latest run type per settlement period
    return (
        df.with_columns(
            pl.col("run_type")
              .replace_strict(RUN_PRECEDENCE, default=0)
              .alias("_run_rank")
        )
        .sort("_run_rank", descending=True)
        .group_by(["settlement_date", "settlement_period"])
        .first()
        .drop("_run_rank")
    )

# atomic write — never leaves a partial file on disk
df.write_parquet(tmp_path)
os.replace(tmp_path, final_path)
Atomic write via os.replace() — no partial files on crash
05
Catalogue registration
DuckDB view auto-created.

After silver writes, init_catalogue() (or refresh_views()) walks the silver and gold directories and creates a DuckDB view per dataset pointing at the Parquet glob. Hive partitioning is enabled so partition columns (year, month) are read as filter predicates — partition pruning is free.

# storage/duckdb.py — view registration
pattern  = "data/silver/elexon/system_prices/**/*.parquet"

con.execute(f"""
    CREATE OR REPLACE VIEW silver_system_prices AS
    SELECT * FROM read_parquet('{pattern}',
                               hive_partitioning=true)
""")
# one view per dataset · all years queryable via SQL
# partition pruning on year=, month= predicates is automatic
All history queryable via one view — no manual union required
06
Query
SQL or Python.

Query via the DuckDB CLI, the GridflowClient Python SDK, or any Parquet-capable tool (pandas, DuckDB WASM, Tableau). The client is read-only and returns Polars DataFrames.

-- System price spread and imbalance · last 7 days
SELECT
    timestamp_utc,
    system_sell_price,
    system_buy_price,
    system_sell_price - system_buy_price AS spread,
    net_imbalance_volume,
    run_type
FROM  silver_system_prices
WHERE settlement_date >= current_date - INTERVAL '7 days'
  AND  run_type = 'DF'   -- final settlement only
ORDER BY timestamp_utc;
from gridflow import GridflowClient

with GridflowClient() as gf:
    prices = gf.get_system_prices(
        start="2026-04-22",
        end="2026-04-29",
    )
    # returns a Polars DataFrame
    spread = prices.with_columns(
        (pl.col("system_sell_price") -
         pl.col("system_buy_price")).alias("spread")
    )
    print(spread.head())
Single DuckDB file — no server, no config, works offline
Rationale

Design decisions.

The choices that shaped the architecture — and the reasoning behind each one.

Why Polars over pandas
Transforms · Silver layer

Polars enforces a strict schema at construction time: you cannot accidentally mix strings and integers in a column the way you can with a pandas object dtype. For a pipeline ingesting seven vendor APIs — each with their own naming conventions and type quirks — that strictness is load-bearing. A cast failure in Polars is a loud exception; in pandas it is often a silent NaN.

The lazy evaluation API also matters. Silver transforms on months of settlement data run entirely in Polars' query planner before any data moves; full scans happen only when .collect() is called. In practice this means a 30-day backfill that would take minutes in pandas runs in seconds.

Why DuckDB as catalogue
Storage · Query layer

DuckDB can query Parquet files directly via read_parquet(glob, hive_partitioning=true) — no ETL step to load data into a running database. The catalogue is a single .duckdb file that holds views, run history, and quality reports; it can be opened read-only from multiple processes simultaneously and checked into a backup without any export step.

The alternative — a Postgres or SQLite database with loaded tables — would require a separate load step after every silver write, and would duplicate storage. DuckDB treats the Parquet files as the source of truth and the .duckdb file as pure metadata. Delete the .duckdb file and gridflow init reconstructs it from the Parquet directory tree in seconds.

Auto-registration pattern
Extensibility · Connectors

Every connector and transformer calls register_connector() or register_transformer() at module level — so the act of importing the module registers it. The CLI imports all source modules at startup, which populates the registry. Adding a new data source means writing the connector and transformer files; no central wiring file needs editing.

This keeps the registry honest: if a connector is not imported, it simply does not exist in the pipeline. There are no configuration files listing available sources — the filesystem is the configuration.

Idempotent backfills
Reliability · Ops

Re-running any ingest or transform command for the same date range produces the same result and leaves the same files on disk. Bronze writes use content-hashed filenames: fetching the same API response twice writes two files with the same name, so the second is a no-op from the filesystem's perspective. Silver writes use os.replace() — an atomic overwrite — so a re-run of the transform replaces the previous Parquet file cleanly.

This means backfill is not a special code path. gridflow transform elexon system_prices --start 2024-01-01 --end 2024-12-31 is safe to run multiple times; it always converges to the correct state.

Settlement revision handling
Domain logic · Elexon

Elexon publishes half-hourly settlement values through up to seven sequential run types: Initial Indicative (II), then Settlement Final (SF), then three reconciliation runs (R1–R3), then Final Reconciliation (RF), then Derived Final (DF). Each run revises the previous. A value marked II today may be materially different when DF publishes 14 days later.

Gridflow's silver transformer keeps only the highest-precedence run per settlement period in the clean layer, but the raw bronze layer retains all runs. Analysts who need the full revision history can read directly from bronze; the silver layer is opinionated toward final settlement for most use cases.

Schema contracts via Pydantic
Data quality · Validation

Every silver dataset has a corresponding Pydantic v2 model. Schema drift — an upstream API adding, removing, or renaming a field — becomes a ValidationError at parse time rather than a silent corruption that propagates downstream. The quality reporter records validation failures to DuckDB, so a field change in the Elexon API on a Tuesday shows up as a failed quality check, not a missing column in a Friday analysis.

The models also serve as the canonical documentation for what each dataset contains. The column names, types, and units in the Pydantic model are the authoritative schema; the dataset pages on this site are generated from them.

Dependencies

The stack.

httpx
Async HTTP client for all seven vendor APIs. The AsyncClient context manager pairs with an asyncio Semaphore to enforce per-source rate limits across concurrent fetches.
polars
DataFrame library for all bronze-to-silver transforms. Strict typing, lazy evaluation, and Parquet I/O that is ~10× faster than pandas for the column-heavy settlement data this pipeline processes.
duckdb
Embedded analytical database. Serves as the catalogue layer — views point at Parquet files; run history and quality reports are stored as tables. No server process, no config.
pydantic v2
Schema validation for every silver dataset. API field mappings are defined as typed models; a schema drift upstream is a loud exception, not a silent null column.
tenacity
Retry policy with exponential backoff for all HTTP calls. Handles transient 429 rate-limit responses and vendor maintenance windows without manual intervention.
typer
CLI framework. Each pipeline operation (ingest, transform, build, init) is a typed command; --last 24h date parsing is handled by a shared utility.
lxml
XML parsing for ENTSO-E responses. The transparency platform API returns large TimeSeries XML documents; lxml handles namespace resolution and element iteration without loading the full document into memory.
pytest + hypothesis
Unit and property-based testing. Transformer tests run against fixture XML/JSON files that mirror real vendor responses, so API changes that break parsing are caught before they reach production data.
uv
Package and virtual-environment manager. Replaces pip + venv + pip-tools with a single lockfile. uv sync from a clean clone reproduces the exact environment in under 10 seconds.

Key abstractions
BaseConnector

Abstract base for all API clients. Defines fetch(dataset, start, end) returning list[RawResponse]. Manages the httpx client lifecycle and rate-limiting semaphore.

BaseSilverTransformer

Template method: run(date) calls read_bronze() → transform() → _write_silver(). Subclasses only implement the source-specific transform logic.

BaseGoldBuilder

Reads from silver Parquet, enriches with derived features, writes partitioned gold Parquet. Gold may also be a plain SQL file registered as a DuckDB cross-source view.

GridflowClient

Read-only DuckDB connection. Returns Polars DataFrames. Wraps common queries (get_system_prices, get_fuel_generation, get_gas_storage) with typed signatures.
