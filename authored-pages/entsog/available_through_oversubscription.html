Overview

What this dataset is.

available_through_oversubscription holds the extra capacity a TSO offers above the firm-technical limit through the oversubscription CMP (Congestion Management Procedure) mechanism. It shares the canonical ENTSO-G operational-data shape with the other indicators; the differentiator is indicator="Available through Oversubscription". Oversubscription is only offered when historical utilisation at a point justifies selling more than the physical technical capacity. Each row is one operator-point-direction's oversubscription capacity for a validity window.

It is sourced from the ENTSO-G Transparency Platform at /api/v1/operationalData with indicator=Available through Oversubscription, queried per operator-point-direction over a gas-day range with timeZone=UCT and periodType=day. The raw JSON lands in bronze and is written to silver via GenericEntsogJsonTransformer (subclassed as AvailableThroughOversubscriptionTransformer), with no Pydantic class — columns are derived dynamically.

As one of the three CMP procedures (oversubscription, surrender, UIOLI), it signals where chronic congestion is letting the TSO sell beyond physical limits — a useful tell for GB interconnection tightness. Pair it with the cmp_* datasets and firm_available for the full congestion picture. Schema and API verified against the gridflow connector source and the 2026-05-08 vault Bronze sample; ENTSO-G is a public, no-authentication API.

SELECT point_key, direction_key,
       MAX_BY(value, period_from) / 1e6 AS extra_gwh
FROM read_parquet('data/silver/entsog/available_through_oversubscription/**/*.parquet')
WHERE period_from <= current_date AND period_to > current_date
  AND value > 0
GROUP BY 1, 2
ORDER BY extra_gwh DESC;

df = pl.read_parquet("data/silver/entsog/available_through_oversubscription/**/*.parquet")
# How often is oversubscription >0?
print(df.group_by("point_key").agg((pl.col("value") > 0).sum().alias("nonzero_rows")).sort("nonzero_rows", descending=True))
Caveats

Things to know before using this.

These are specific to available_through_oversubscription. Also read the ENTSO-G-wide caveats on the vendor page.

01
Indicator string is exact-case (mixed case)

Connector sends the literal "Available through Oversubscription" — note the lowercase through. OPERATIONAL_INDICATORS["available_through_oversubscription"] · connectors/entsog/endpoints.py L111

02
Most rows are zero

Oversubscription only kicks in when historical congestion justifies it. Empty (zero-value) rows are the norm at uncongested points. domain knowledge / CMP mechanism docs

03
timeZone=UCT (vendor typo, intentional)

Connector sends UCT not UTC; sending UTC returns HTTP 400. connectors/entsog/endpoints.py L17 · ENTSOG_TIMEZONE

04
Empty windows return HTTP 404, not 200

ENTSO-G's empty convention is HTTP 404 + {"message":"No result found"}. Connector short-circuits as empty-bronze. connectors/entsog/client.py L24-28, L109-115

05
Part of CMP — pair with cmp_* datasets

Oversubscription is one of three CMP procedures (oversubscription, surrender, UIOLI). For the full CMP picture, also query cmp_unsuccessful_requests, cmp_unavailable_firm_capacity, cmp_auction_premiums. ENTSO-G Network Code on CAM/CMP
