Overview

What this dataset is.

nominations holds shipper-submitted day-ahead nominations per operator-point-direction — the intended gas quantities each shipper plans to move through a connection point for the coming gas day. It is the canonical pre-allocation signal for gas-supply tightness and CCGT dispatch likelihood, queried at the GB interconnection points (Bacton IUK, Bacton BBL, Moffat). Each row is one operator-point-direction's nominated quantity for a single gas day, reported in kWh/day.

It is sourced from the ENTSO-G Transparency Platform at /api/v1/operationalData?indicator=Nomination, queried per operator-point-direction over a gas-day range with timeZone=UCT. The raw JSON lands in bronze and is written to the silver parquet partition via GenericEntsogJsonTransformer, with no Pydantic class — the column shape is derived dynamically from the live response rather than validated against a fixed schema.

As the day-ahead intent signal, nominations precede the realised numbers in physical_flows and allocations — the nomination-vs-actual delta is the canonical short-term read on gas-supply tightness and gas-fired dispatch. Schema and API verified against the gridflow connector source and the 2026-05-08 vault Bronze sample; ENTSO-G is a public, no-authentication API.

-- 30-day Bacton exit nominations vs actual physical flow (join to physical_flows)
SELECT n.period_from::date AS gas_day,
       n.point_key,
       n.value / 1e6 AS nominated_gwh,
       p.flow_gwh_per_day AS actual_gwh
FROM read_parquet('data/silver/entsog/nominations/**/*.parquet') n
JOIN read_parquet('data/silver/entsog/physical_flows/**/*.parquet') p
  ON  n.period_from = p.timestamp_utc
  AND n.point_key   = p.point_key
  AND n.direction_key = p.direction_key
WHERE n.point_key = 'ITP-00005'
  AND n.direction_key = 'exit'
  AND n.period_from >= current_date - INTERVAL 30 DAY
ORDER BY 1;
import polars as pl

df = pl.read_parquet("data/silver/entsog/nominations/**/*.parquet")
# Daily aggregate nomination by GB direction
gb = (
    df.filter(pl.col("operator_key").str.starts_with("UK-TSO-"))
      .group_by(["period_from", "direction_key"])
      .agg((pl.col("value").sum() / 1e6).alias("gwh"))
      .sort("period_from")
)
print(gb.tail(14))
Caveats

Things to know before using this.

These are specific to nominations. Also read the ENTSO-G-wide caveats on the vendor page.

01
Indicator string is exact-case

Vendor rejects nomination / Nominations; connector sends literal "Nomination". vault Known Issues · OPERATIONAL_INDICATORS["nominations"] = "Nomination" · connectors/entsog/endpoints.py L97

02
timeZone=UCT (vendor typo, intentional)

Connector sends UCT not UTC; sending UTC returns HTTP 400. connectors/entsog/endpoints.py L17 · ENTSOG_TIMEZONE

03
Empty windows return HTTP 404, not 200

ENTSO-G's empty convention is HTTP 404 + {"message":"No result found"}. Connector short-circuits as empty-bronze. connectors/entsog/client.py L24-28, L109-115

04
Nominations precede allocations

A Nomination row is a shipper request; the eventual delivery sits in Allocation and Physical Flow. Join all three on (timestamp_utc, point_key, operator_key, direction_key) for nomination-vs-actual analysis. vault Modelling notes · ENTSO-G data dictionary

05
flow_status is typically empty for nominations

Unlike physical_flows (which carries Provisional/Confirmed), nominations are a single point-in-time submission and the flow_status field is generally empty. vault Bronze sample L117

06
Period offset is CEST/CET even with timeZone=UCT

Despite requesting UCT, periodFrom/periodTo carry +02:00 / +01:00. parse_entsog_datetime_expr converts to UTC at silver. silver/entsog/datetime.py · vault Known Issues
