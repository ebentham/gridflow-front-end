Overview

What this dataset is.

actual_generation holds realised megawatt output per EIC bidding zone, broken down by Primary Source Resource (PSR) production type — coal, gas, wind onshore, wind offshore, solar, hydro, nuclear, biomass and the other categories from the B01..B25 EIC codelist. Each row is one production type's contribution to a single 15-, 30-, or 60-minute settlement interval.

It is sourced from the ENTSO-E Transparency Platform at /api?documentType=A75&processType=A16, queried per bidding zone over a date range. The raw XML lands in bronze as a GL_MarketDocument, is validated against EntsoeActualGeneration, and written to the silver parquet partition via ActualGenerationTransformer.

This is the canonical residual-load input for European power-market models and the cross-vendor analogue of GB's fuelhh published by individual TSOs. Schema verified against gridflow source on 2026-05-20; the live API requires extended ENTSO-E registration.

-- Daily generation mix for DE-LU last 7 days
SELECT date_trunc('day', timestamp_utc) AS day,
       production_type,
       sum(generation_mw) / count(*) AS avg_mw
FROM read_parquet('data/silver/entsoe/actual_generation/**/*.parquet')
WHERE area_code = '10Y1001A1001A82H'
  AND timestamp_utc >= current_timestamp - INTERVAL 7 DAY
GROUP BY 1, 2
ORDER BY 1, 2;
import polars as pl

df = pl.read_parquet(
    "data/silver/entsoe/actual_generation/**/*.parquet",
)
# Pivot to wide: one column per production type
mix = df.filter(pl.col("area_code") == "10Y1001A1001A82H").pivot(
    index="timestamp_utc",
    on="production_type",
    values="generation_mw",
    aggregate_function="sum",
)
# Residual load = actual_load - wind - solar (canonical MEF feature)
print(mix.select(["timestamp_utc", "B19", "B18", "B16"]).tail())
Caveats

Things to know before using this.

These are specific to actual_generation. Also read the ENTSO-E-wide caveats on the vendor page.

01
GB stopped publishing post-Brexit

Calls against GB (10YGB----------A) return Acknowledgement_MarketDocument reason code 999 — no data. Use Elexon fuelhh / fuelinst for the GB generation mix instead. vault Known Issues #1

02
Resolution varies by zone

DE-LU, AT, FR publish at PT15M; most other zones at PT60M. Don't assume hourly resolution when joining against settlement-period datasets — either filter on resolution or aggregate PT15M → PT60M before the join. vault Known Issues #2

03
Revisions silently overwrite

The same (timestamp_utc, area_code, production_type) tuple may be republished as TSOs revise their figures; the transformer keeps the latest on sort-last. No published_at column is surfaced — point-in-time queries need to look at the bronze layer. silver/entsoe/actual_generation.py L72-74

04
DISCREPANCY · schema vs transformer
area_name declared but not emitted

The schema declares area_name: str = "" (schemas/entsoe.py L51) but the transformer's output_cols omits it — so the silver parquet has no area_name column at all. Downstream code that needs human-readable zone names must join an EIC reference. schemas/entsoe.py L51 · silver/entsoe/actual_generation.py L82-87

05
Storage rows can be negative

A75 includes both generation (businessType=A03) and consumption (A04) for storage classes. B10 (Hydro Pumped Storage) and B12 (Hydro Water Reservoir) rows can carry negative MW when the unit is charging from the grid. Filter or sign-handle before computing net-generation totals. vault Known Issues #5

06
Entitlement-blocked on the default API key

The live API returns HTTP 401 for the unregistered gridflow default key — A75 needs extended ENTSO-E registration. The schema, transformer, and sample below are verified from gridflow source; live verification is deferred until the registration lands. .planning/reconciliation/entsoe/01-actual-generation-http-401.md
