Overview

What this dataset is.

load_forecast_monthly holds the month-ahead TSO load forecast in megawatts per EIC bidding zone. The Pydantic class is the shared EntsoeLoadForecast — same fields as the day-ahead surface — and the surface is distinguished by the forecast_horizon constant, which the subclass transformer pins to "month_ahead". Each row is one bidding zone's monthly-view forecast for a single hourly settlement interval.

It is sourced from the ENTSO-E Transparency Platform at /api?documentType=A65&processType=A32, queried per bidding zone over a monthly window using the outBiddingZone_Domain param style. The raw XML lands in bronze, is validated against the shared EntsoeLoadForecast class, and written to the silver parquet partition via LoadForecastMonthlyTransformer.

This is the monthly capacity-adequacy view — useful for gas-power balance modelling and as the medium-horizon term in a term-structure stack alongside the day-ahead and yearly forecasts. Month-ahead values are smooth — daily peak variation is muted relative to day-ahead, because the forecast represents an averaged expectation rather than a date-specific call. Verified against vendor docs on 2026-05-08; A65/A32 is one of the ENTSO-E endpoints accessible with the free gridflow default API key.

-- Per-zone monthly mean forecast (last 12 months)
SELECT date_trunc('month', timestamp_utc) AS month, area_code,
       avg(load_forecast_mw) AS mean_mw
FROM read_parquet('data/silver/entsoe/load_forecast_monthly/**/*.parquet')
WHERE timestamp_utc >= current_timestamp - INTERVAL 12 MONTH
GROUP BY 1, 2
ORDER BY 1, 2;
import polars as pl

mfc = pl.read_parquet("data/silver/entsoe/load_forecast_monthly/**/*.parquet")
yfc = pl.read_parquet("data/silver/entsoe/load_forecast_yearly/**/*.parquet")
# Monthly revision of the yearly forecast
diff = mfc.join(yfc, on=["timestamp_utc", "area_code"], suffix="_y").with_columns(
    (pl.col("load_forecast_mw") - pl.col("load_forecast_mw_y")).alias("revision_mw")
)
print(diff.group_by("area_code").agg(pl.col("revision_mw").mean()))
Caveats

Things to know before using this.

These are specific to load_forecast_monthly. Also read the ENTSO-E-wide caveats on the vendor page.

01
Reuses load_forecast Pydantic schema

The Pydantic class is EntsoeLoadForecast (shared with load_forecast); the difference is the forecast_horizon constant set by the subclass transformer. Always filter on forecast_horizon when reading combined load_forecast_* datasets. silver/entsoe/load_forecast_monthly.py

02
outBiddingZone_Domain (not in_Domain)

A65/A32 uses outBiddingZone_Domain. endpoints.py L126-128

03
Monthly forecast is smooth

Month-ahead forecasts represent averaged expectations — daily peak variation is muted relative to day-ahead. Don't expect day-ahead-quality calibration. domain knowledge

04
Use a monthly window

Short windows (<1 month) may return EMPTY or partial. Build period bounds at month boundaries. vault Known Issues

05
Not entitlement-blocked

A65/A32 is one of the 14 ENTSO-E endpoints accessible with the free gridflow default API key. .planning/reconciliation/entsoe/ — no load_forecast_monthly-http-401.md
