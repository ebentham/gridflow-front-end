Overview

What this dataset is.

congestion_management_costs is the cost a TSO incurs to resolve grid congestion — redispatching and countertrading outlays — reported in EUR per zone under DocumentType A92. It is the cost-side counterpart to congestion_income (A25/B10): where income is the rent collected from price spreads, this is what the operator pays to keep flows within limits. Each row carries (timestamp_utc, in_area_code, out_area_code, business_type) as the primary key and amount_eur as the value — though uniquely among the H6 amount datasets it is single-zone scoped.

It is sourced from the ENTSO-E Transparency Platform at /api?documentType=A92&in_Domain={EIC}&out_Domain={EIC}, queried per zone. The raw XML lands in bronze and is validated against the shared H6 amount class EntsoeTransmissionMarketAmount (schemas/entsoe.py L577-594), then written to silver via CongestionManagementCostsTransformer (value_tag="price.amount"). Because its connector declares domain_style="zone" while the H6 amount schema carries both in/out area codes, out_area_code may equal in_area_code or be empty — surfaced as a discrepancy in the caveats below.

It is the headline number behind German Engpassmanagement and the EU congestion-cost debate: read against congestion_income it gives a net P&L (consistently negative for DE-LU, positive for transit zones), and the MWh behind the cost is in redispatching_internal, redispatching_cross_border, and countertrading. Schema verified against gridflow source on 2026-05-20; the live API requires extended ENTSO-E registration.

-- Monthly congestion costs by zone (last 12 months)
SELECT date_trunc('month', timestamp_utc) AS month,
       in_area_code AS zone,
       sum(amount_eur) AS monthly_cost_eur
FROM read_parquet('data/silver/entsoe/congestion_management_costs/**/*.parquet')
WHERE timestamp_utc >= current_timestamp - INTERVAL 365 DAY
GROUP BY 1, 2 ORDER BY 1 DESC, monthly_cost_eur DESC;
import polars as pl

cost = pl.read_parquet("data/silver/entsoe/congestion_management_costs/**/*.parquet")
income = pl.read_parquet("data/silver/entsoe/congestion_income/**/*.parquet")
# Net congestion P&L per zone (income - costs)
net = income.group_by(["timestamp_utc", "in_area_code"]).agg(
    pl.col("amount_eur").sum().alias("income_eur")
).join(
    cost.group_by(["timestamp_utc", "in_area_code"]).agg(
        pl.col("amount_eur").sum().alias("cost_eur")
    ),
    on=["timestamp_utc", "in_area_code"], how="outer"
)
print(net.with_columns(
    (pl.col("income_eur").fill_null(0) - pl.col("cost_eur").fill_null(0)).alias("net_eur")
).tail())
Caveats

Things to know before using this.

One item below is a data-source discrepancy — the connector's domain style and the shared H6 amount schema disagree on scope. Surfaced so downstream consumers interpret the area codes correctly. Also read the ENTSO-E-wide caveats.

01
Data-source discrepancy · domain_style
domain_style="zone" — not zone-pair

The connector declares domain_style="zone" (endpoints.py L184), but the shared H6 amount schema EntsoeTransmissionMarketAmount carries both in_area_code and out_area_code. This is the only H6 amount dataset with single-zone scope: query with in_Domain only; out_area_code may equal in_area_code or be empty. endpoints.py L184 · frontmatter discrepancy

02
Pairs with congestion_income for net P&L

Cost-side counterpart to A25/B10. Net = income − cost; consistently negative for DE-LU (high redispatch costs) and positive for transit zones. domain knowledge

03
H6 amount transformer

Uses H6 amount-transformer variant. Output column is amount_eur. silver/entsoe/h6_market.py L121-124

04
Revisions overwrite

Same (timestamp_utc, in_area_code, out_area_code, business_type) overwrites silently. silver/entsoe/h6_market.py L85-91

05
Entitlement-blocked on default API key

Live API returns HTTP 401 for the unregistered gridflow default key. Schema verified from gridflow source. .planning/reconciliation/entsoe/08-congestion-management-costs-http-401.md
