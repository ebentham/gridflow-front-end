Overview

What this dataset is.

activated_balancing_qty holds the activated balancing-energy quantity in MWh per control area, split by reserve type and direction. The transformer maps businessType codes A95 / A96 / A97 / A98 to reserve_type fcr / afrr / mfrr / rr, and flowDirection A01 / A02 to direction up / down. Each row is one reserve type × direction's activated MWh for a single activation interval.

It is sourced from the ENTSO-E Transparency Platform at /api?documentType=A83&processType=A16. Unlike sibling datasets, activated_balancing_qty has no entry in the connector's DOC_TYPES registry, so callers must dispatch the A83/A16 query manually via **params injection. The raw XML lands in bronze, is validated against EntsoeActivatedBalancingQty, and written to the silver parquet partition via ActivatedBalancingQtyTransformer.

Pair on the full (timestamp_utc, area_code, reserve_type, direction) key with activated_balancing_prices for activation revenue (€/MWh × MWh), with contracted_reserves for utilisation rates (activated MWh / contracted MW), and with current_balancing_state to relate the area control error to the resulting up- vs down-activation mix. Verified against vendor docs on 2026-05-08; A83/A16 is accessible with the free gridflow default API key.

-- Activation revenue per reserve_type per zone (last 30 days)
SELECT q.area_code, q.reserve_type, q.direction,
       sum(q.quantity_mwh) AS total_mwh,
       sum(q.quantity_mwh * p.price_eur_mwh) AS revenue_eur
FROM read_parquet('data/silver/entsoe/activated_balancing_qty/**/*.parquet') q
JOIN read_parquet('data/silver/entsoe/activated_balancing_prices/**/*.parquet') p
  USING (timestamp_utc, area_code, reserve_type, direction)
WHERE q.timestamp_utc >= current_timestamp - INTERVAL 30 DAY
GROUP BY 1, 2, 3 ORDER BY revenue_eur DESC NULLS LAST;
import polars as pl

qty = pl.read_parquet("data/silver/entsoe/activated_balancing_qty/**/*.parquet")
# Daily activation volume by reserve_type
daily = qty.with_columns(pl.col("timestamp_utc").dt.date().alias("day")).group_by(
    ["day", "reserve_type", "direction"]
).agg(pl.col("quantity_mwh").sum())
print(daily.tail())
Caveats

Things to know before using this.

These are specific to activated_balancing_qty. Also read the ENTSO-E-wide caveats on the vendor page.

01
DISCREPANCY · schema vs connector
No DOC_TYPES entry in endpoints.py

activated_balancing_qty has no entry in connectors/entsoe/endpoints.py::DOC_TYPES. Callers must dispatch the A83/A16 query manually via **params injection. Finding 36-activated-balancing-qty-manual-transformer-schema.md (wontfix v3-candidate) flagged the missing class — now closed by the L384-407 addition, but the connector dispatch remains manual. connector source · schemas/entsoe.py L384-407 · .planning/reconciliation/entsoe/36-activated-balancing-qty-manual-transformer-schema.md

02
reserve_type code mapping

businessType A95/A96/A97/A98 → reserve_type "fcr"/"afrr"/"mfrr"/"rr" via replace_strict. silver/entsoe/activated_balancing_qty.py L68-70

03
quantity_mwh (MWh) not quantity_mw

Unit is MWh — the integrated energy over the activation interval. Distinct from MW (instantaneous power) in capacity datasets. schemas/entsoe.py L396

04
Both directions surface separately

Up and down activations have separate rows. To compute net activation, sign-handle: up − down. schemas/entsoe.py L395

05
Not entitlement-blocked

A83/A16 is accessible with the free gridflow default API key. .planning/reconciliation/entsoe/ — no activated_balancing_qty-http-401.md
