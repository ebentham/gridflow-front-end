Overview

What this dataset is.

countertrading is the hourly countertrading volume in MW per directed zone-pair under DocumentType A91 — the reverse-direction trade a TSO buys on the market to relieve a congested border. It differs from redispatching in mechanism: countertrading clears through the market (the TSO buys reverse capacity at market prices) rather than directing specific plants up or down. Each row carries (timestamp_utc, in_area_code, out_area_code, business_type) as the primary key and quantity_mw as the value.

It is sourced from the ENTSO-E Transparency Platform at /api?documentType=A91&in_Domain={EIC}&out_Domain={EIC}, queried per directed border-pair. The raw XML lands in bronze and is validated against the shared H6 quantity class EntsoeTransmissionMarketQuantity (schemas/entsoe.py L557-574), then written to silver via CountertradingTransformer. The H6 quantity class is shared across twelve transmission datasets, so the column shape is identical to redispatching — selection is by dataset key, not schema.

It is one of the two congestion-relief levers a TSO can pull: read against redispatching_cross_border it shows the mechanism split per border, and its monetary footprint surfaces in congestion_management_costs (A92). Countertrading activity also tends to track reductions in net_transfer_capacity. Schema verified against gridflow source on 2026-05-20; the live API requires extended ENTSO-E registration.

-- Countertrading activity by border (last 90 days)
SELECT in_area_code, out_area_code,
       count(*) FILTER (WHERE quantity_mw > 0) AS active_hours,
       sum(quantity_mw) AS total_mwh
FROM read_parquet('data/silver/entsoe/countertrading/**/*.parquet')
WHERE timestamp_utc >= current_timestamp - INTERVAL 90 DAY
GROUP BY 1, 2
ORDER BY total_mwh DESC;
import polars as pl

ct = pl.read_parquet("data/silver/entsoe/countertrading/**/*.parquet")
rd = pl.read_parquet("data/silver/entsoe/redispatching_cross_border/**/*.parquet")
# Countertrade vs redispatch — alternative congestion mechanisms by border
ct_sum = ct.group_by(["in_area_code", "out_area_code"]).agg(pl.col("quantity_mw").sum().alias("ct_mwh"))
rd_sum = rd.group_by(["in_area_code", "out_area_code"]).agg(pl.col("quantity_mw").sum().alias("rd_mwh"))
print(ct_sum.join(rd_sum, on=["in_area_code", "out_area_code"], how="outer"))
Caveats

Things to know before using this.

These are specific to countertrading. Also read the ENTSO-E-wide caveats on the vendor page.

01
Alternative to redispatching

Countertrading and redispatching are two TSO mechanisms for managing congestion — countertrading uses market clearing (the TSO buys reverse capacity at market prices), redispatching uses direct generation control. domain knowledge

02
Directional per border-pair

Each border-pair has two directions; pair both for the full picture. endpoints.py L174-179

03
Pydantic class is shared (H6 family)

EntsoeTransmissionMarketQuantity covers 12 H6 datasets. Finding 47-countertrading-no-silver-schema-table.md (wontfix v3-candidate) is stale. schemas/entsoe.py L557

04
Revisions overwrite

Same (timestamp_utc, in_area_code, out_area_code, business_type) overwrites silently. silver/entsoe/h6_market.py L85-91

05
Entitlement-blocked on default API key

Live API returns HTTP 401 for the unregistered gridflow default key. Schema verified from gridflow source. .planning/reconciliation/entsoe/09-countertrading-http-401.md
