Overview

What this dataset is.

load_forecast_weekly holds the week-ahead TSO load forecast in megawatts per EIC bidding zone. The Pydantic schema is structurally identical to its day-ahead sibling load_forecast; the difference is the forecast_horizon constant — pinned to "week_ahead" by this surface's transformer. Each row is one bidding zone's week-ahead forecast for a single hourly settlement interval, covering the upcoming Monday-to-Sunday week.

It is sourced from the ENTSO-E Transparency Platform at /api?documentType=A65&processType=A31, queried per bidding zone over a weekly window using the outBiddingZone_Domain param style. The raw XML lands in bronze, is validated against EntsoeLoadForecastWeekly, and written to the silver parquet partition via LoadForecastWeeklyTransformer.

This is the short-term capacity-adequacy view — useful for stress tests that the 24-hour day-ahead surface cannot support. Stack against load_forecast for the same hours to measure how much the closer-in day-ahead forecast revises the week-ahead view, and against actual_load for week-ahead forecast skill. Verified against vendor docs on 2026-05-08; A65/A31 is one of the ENTSO-E endpoints accessible with the free gridflow default API key.

-- Week-ahead vs day-ahead vs actual for the same hours (term-structure test)
SELECT a.timestamp_utc, a.area_code,
       w.load_forecast_mw AS week_ahead,
       d.load_forecast_mw AS day_ahead,
       a.load_mw          AS actual
FROM read_parquet('data/silver/entsoe/actual_load/**/*.parquet') a
LEFT JOIN read_parquet('data/silver/entsoe/load_forecast_weekly/**/*.parquet') w
  ON a.timestamp_utc = w.timestamp_utc AND a.area_code = w.area_code
LEFT JOIN read_parquet('data/silver/entsoe/load_forecast/**/*.parquet') d
  ON a.timestamp_utc = d.timestamp_utc AND a.area_code = d.area_code
WHERE a.timestamp_utc >= current_timestamp - INTERVAL 30 DAY
ORDER BY a.timestamp_utc DESC LIMIT 50;
import polars as pl

week = pl.read_parquet("data/silver/entsoe/load_forecast_weekly/**/*.parquet")
day = pl.read_parquet("data/silver/entsoe/load_forecast/**/*.parquet")
# Forecast-revision: how much does the day-ahead update the week-ahead?
diff = week.join(day, on=["timestamp_utc", "area_code"], suffix="_da").with_columns(
    (pl.col("load_forecast_mw_da") - pl.col("load_forecast_mw")).alias("revision_mw")
)
print(diff.group_by("area_code").agg(pl.col("revision_mw").abs().mean()))
Caveats

Things to know before using this.

These are specific to load_forecast_weekly. Also read the ENTSO-E-wide caveats on the vendor page.

01
forecast_horizon = "week_ahead" — not day_ahead

This dataset and load_forecast share an identical Pydantic schema except for the forecast_horizon default. Always filter on forecast_horizon when combining horizons. silver/entsoe/load_forecast_weekly.py L73

02
outBiddingZone_Domain (not in_Domain)

A65/A31 uses outBiddingZone_Domain. endpoints.py L123-125

03
Use a weekly window

Week-ahead forecast — short windows may return EMPTY or partial. Build period bounds at Mon–Mon boundaries. vault Known Issues

04
Forecast revisions overwrite

Same (timestamp_utc, area_code) re-publication overwrites silently on dedup. No published_at for PIT — to track week-ahead vs day-ahead deltas, snapshot the silver partition. silver/entsoe/load_forecast_weekly.py L67

05
Not entitlement-blocked

A65/A31 is one of the 14 ENTSO-E endpoints accessible with the free gridflow default API key. .planning/reconciliation/entsoe/ — no load_forecast_weekly-http-401.md
