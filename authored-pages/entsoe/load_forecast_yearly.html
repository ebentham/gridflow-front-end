Overview

What this dataset is.

load_forecast_yearly holds the year-ahead TSO load forecast in megawatts per EIC bidding zone, covering all 8760 hours of a forward calendar year. The Pydantic class is the shared EntsoeLoadForecast — same fields as day-ahead — with the subclass transformer overriding forecast_horizon to "year_ahead". Each row is one bidding zone's year-ahead forecast for a single hourly settlement interval.

It is sourced from the ENTSO-E Transparency Platform at /api?documentType=A65&processType=A33, queried per bidding zone over a year-bounded window using the outBiddingZone_Domain param style. The raw XML lands in bronze, is validated against the shared EntsoeLoadForecast class, and written to the silver parquet partition via LoadForecastYearlyTransformer.

This is the structural baseline that the closer-in load_forecast, weekly, and monthly surfaces refine — and the demand side of the canonical forecast_margin (A70/A33) calculation against generation_forecast. Verified against vendor docs on 2026-05-08; A65/A33 is one of the ENTSO-E endpoints accessible with the free gridflow default API key.

-- Year-ahead vs realised: monthly mean comparison
SELECT date_trunc('month', y.timestamp_utc) AS month, y.area_code,
       avg(y.load_forecast_mw) AS year_ahead_mean,
       avg(a.load_mw) AS actual_mean,
       avg(a.load_mw) - avg(y.load_forecast_mw) AS error_mw
FROM read_parquet('data/silver/entsoe/load_forecast_yearly/**/*.parquet') y
JOIN read_parquet('data/silver/entsoe/actual_load/**/*.parquet') a
  ON y.timestamp_utc = a.timestamp_utc AND y.area_code = a.area_code
GROUP BY 1, 2
ORDER BY 1, 2;
import polars as pl

yfc = pl.read_parquet("data/silver/entsoe/load_forecast_yearly/**/*.parquet")
gfc = pl.read_parquet("data/silver/entsoe/generation_forecast/**/*.parquet")
# Forecast margin (DIY) — should match the published forecast_margin dataset
margin = gfc.group_by(["timestamp_utc", "area_code"]).agg(
    pl.col("generation_forecast_mw").sum().alias("gen_mw")
).join(yfc, on=["timestamp_utc", "area_code"]).with_columns(
    (pl.col("gen_mw") - pl.col("load_forecast_mw")).alias("diy_margin_mw")
)
print(margin.tail())
Caveats

Things to know before using this.

These are specific to load_forecast_yearly. Also read the ENTSO-E-wide caveats on the vendor page.

01
Reuses load_forecast Pydantic schema

Same EntsoeLoadForecast class as load_forecast; only the forecast_horizon constant differs (subclass override). Filter on forecast_horizon when combining horizons. silver/entsoe/load_forecast_yearly.py

02
outBiddingZone_Domain (not in_Domain)

A65/A33 uses outBiddingZone_Domain. endpoints.py L129-131

03
Use a yearly window

Year-ahead forecast — short windows may return EMPTY. Build period bounds at year boundaries. vault Known Issues

04
Pair with generation_forecast for margin

generation_forecast − load_forecast_yearly is the conceptual basis for the forecast_margin (A70/A33) dataset. forecast_margin.md brief

05
Not entitlement-blocked

A65/A33 is one of the 14 ENTSO-E endpoints accessible with the free gridflow default API key. .planning/reconciliation/entsoe/ — no load_forecast_yearly-http-401.md
