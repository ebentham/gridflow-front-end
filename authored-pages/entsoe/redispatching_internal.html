Overview

What this dataset is.

redispatching_internal is the within-zone redispatching volume in MW — generation the TSO shifts around inside a single bidding zone to keep internal grid elements within limits. It shares DocumentType A63 with redispatching_cross_border, distinguished by businessType=A85 (internal) versus A46 (cross-border). Each row carries (timestamp_utc, in_area_code, out_area_code, business_type) as the primary key and quantity_mw as the value — and for internal redispatch in_area_code == out_area_code.

It is sourced from the ENTSO-E Transparency Platform at /api?documentType=A63&businessType=A85&in_Domain={EIC}&out_Domain={EIC}, queried per zone (both domain params set to the same EIC). The raw XML lands in bronze and is validated against the shared H6 quantity class EntsoeTransmissionMarketQuantity (schemas/entsoe.py L557-574), then written to silver via RedispatchingInternalTransformer. The "from / to" sense of an internal redispatch (which sub-zone moves up versus down) is not encoded in the silver schema.

It is the headline congestion-management metric for Germany: midday solar peaks routinely force massive north-to-south redispatch from wind in the north to load centres in the south. The MWh here drives the EUR in congestion_management_costs (A92), and the distortion between scheduled and realised dispatch is recoverable by comparing actual_generation against generation_forecast. Schema verified against gridflow source on 2026-05-20; the live API requires extended ENTSO-E registration.

-- Internal redispatching cost per zone (last 30 days)
SELECT date_trunc('day', r.timestamp_utc) AS day, r.in_area_code AS zone,
       sum(r.quantity_mw) AS total_mwh,
       sum(c.amount_eur) AS total_cost_eur
FROM read_parquet('data/silver/entsoe/redispatching_internal/**/*.parquet') r
LEFT JOIN read_parquet('data/silver/entsoe/congestion_management_costs/**/*.parquet') c
  ON r.timestamp_utc = c.timestamp_utc AND r.in_area_code = c.in_area_code
WHERE r.timestamp_utc >= current_timestamp - INTERVAL 30 DAY
GROUP BY 1, 2
ORDER BY total_cost_eur DESC NULLS LAST;
import polars as pl

rd = pl.read_parquet("data/silver/entsoe/redispatching_internal/**/*.parquet")
# Hours with significant internal redispatching by zone
heavy = rd.filter(pl.col("quantity_mw") > 1000).group_by("in_area_code").agg(
    pl.len().alias("heavy_hours"),
    pl.col("quantity_mw").mean().alias("mean_mw")
)
print(heavy.sort("heavy_hours", descending=True))
Caveats

Things to know before using this.

These are specific to redispatching_internal. Also read the ENTSO-E-wide caveats on the vendor page.

01
in_area_code == out_area_code

For internal redispatching, both domain params are the same zone EIC. The "from / to" interpretation is the internal sub-zone direction, encoded outside the silver schema. vault Known Issues · endpoints.py L167-173

02
A63 shared with redispatching_cross_border (businessType A46)

Select on dataset key, not DocumentType alone. endpoints.py L160-173

03
Pydantic class is shared (H6 family)

EntsoeTransmissionMarketQuantity covers 12 H6 datasets. Finding 68-redispatching-internal-no-table.md (wontfix v3-candidate) is stale. schemas/entsoe.py L557

04
Revisions overwrite

Same (timestamp_utc, in_area_code, out_area_code, business_type) overwrites silently. silver/entsoe/h6_market.py L85-91

05
Entitlement-blocked on default API key

Live API returns HTTP 401 for the unregistered gridflow default key. Schema verified from gridflow source. .planning/reconciliation/entsoe/30-redispatching-internal-http-401.md
