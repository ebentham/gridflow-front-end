Overview

What this dataset is.

imbalance_volume holds the EU imbalance volume in MWh per control area, split into long (generation excess) and short (consumption excess) directions. The transformer maps flowDirection codes A01 / A02 to direction long / short via replace_strict — note this is opposite to imbalance_prices, where businessType (A19/A20) drives direction. Each row is one direction's volume for a single 30-minute interval.

It is sourced from the ENTSO-E Transparency Platform at /api?documentType=A86&businessType=A19, queried per control area over a date range. The connector pins businessType=A19; the direction column is derived from flowDirection on each <TimeSeries>. The raw XML lands in bronze, is validated against EntsoeImbalanceVolume, and written to the silver parquet partition via ImbalanceVolumeTransformer.

Pair with imbalance_prices on (timestamp_utc, area_code, direction) for the settlement value (€ = volume × price), with current_balancing_state as the sister A86 surface (B33 businessType — same magnitude, different framing), and with activated_balancing_qty because the activated volume closes the imbalance gap. The GB-native equivalent is Elexon's NIV (net_imbalance_volume). Verified against vendor docs on 2026-05-08; A86 (with businessType=A19) is accessible with the free gridflow default API key.

-- Net imbalance per zone per period (signed: positive=short)
SELECT timestamp_utc, area_code,
       sum(CASE WHEN direction='short' THEN volume_mwh ELSE 0 END) -
       sum(CASE WHEN direction='long' THEN volume_mwh ELSE 0 END) AS net_imbalance_mwh
FROM read_parquet('data/silver/entsoe/imbalance_volume/**/*.parquet')
WHERE timestamp_utc >= current_timestamp - INTERVAL 7 DAY
GROUP BY 1, 2 ORDER BY abs(net_imbalance_mwh) DESC LIMIT 50;
import polars as pl

iv = pl.read_parquet("data/silver/entsoe/imbalance_volume/**/*.parquet")
ip = pl.read_parquet("data/silver/entsoe/imbalance_prices/**/*.parquet")
# Imbalance settlement value (€) = volume × price per direction
val = iv.join(ip, on=["timestamp_utc", "area_code", "direction"]).with_columns(
    (pl.col("volume_mwh") * pl.col("price_eur_mwh")).alias("settlement_eur")
)
print(val.group_by(["area_code", "direction"]).agg(pl.col("settlement_eur").sum()))
Caveats

Things to know before using this.

These are specific to imbalance_volume. Also read the ENTSO-E-wide caveats on the vendor page.

01
businessType A19 pinned, flowDirection drives direction

Connector pins businessType=A19; the direction field is derived from flowDirection (A01/A02), not businessType. This is opposite to imbalance_prices, where businessType (A19/A20) drives direction. endpoints.py L299-300 · silver/entsoe/imbalance_volume.py L63-66

02
volume_mwh not volume_mw

Unit is MWh (integrated over interval). Distinct from MW (instantaneous) elsewhere. schemas/entsoe.py L371

03
Sign of short/long depends on TSO convention

ENTSO-E convention: long = system surplus (generation excess), short = system deficit. Some downstream models use opposite signs. domain knowledge

04
Revisions overwrite

Imbalance settlement is revised over time — keep="last" keeps the most recent. silver/entsoe/imbalance_volume.py L75

05
Not entitlement-blocked

A86 (with businessType=A19) is accessible with the free gridflow default API key. .planning/reconciliation/entsoe/ — no imbalance_volume-http-401.md
