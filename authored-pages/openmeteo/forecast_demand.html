Overview

What this dataset is.

forecast_demand holds the hourly weather that drives GB electricity demand — air temperature, wind, humidity, surface pressure, precipitation and snow — forecast 1–16 days ahead at seven population-weighted GB cities. It is the forward-looking half of Gridflow's demand-weather pair, drawn from Open-Meteo's numerical-weather-prediction (NWP) router rather than the ERA5 archive. Each row is one hour at one GB site, carrying the raw met variables plus the derived heating- and cooling-degree-hour features the load model consumes.

It is sourced from Open-Meteo's forecast endpoint at api.open-meteo.com/v1/forecast, queried once per city, validated against DemandWeather (schemas/weather.py), and written to silver via ForecastDemandWeather. The payload is JSON columnar — parallel hourly arrays (hourly.time[], hourly.temperature_2m[], …) that the transformer zips into one row per timestamp before the F15-B unit-suffix rename.

Demand weather is the single largest driver of short-run GB load, and this feed is the day-ahead input to that model: the HDD/CDD features here pair with realised demand from Elexon indo / fuelhh to score forecast skill, while the ERA5 sibling historical_demand supplies the multi-decade calibration backtest. -- Day-ahead GB demand-weighted temperature (population-weighted by city)
SELECT date_trunc('hour', timestamp_utc) AS hour,
       sum(temperature_2m_c * pop_weight) AS gb_pop_temp_c,
       sum(hdd_k * pop_weight)            AS gb_pop_hdd_k
FROM read_parquet('data/silver/open_meteo/forecast_demand/**/*.parquet') f
JOIN (VALUES ('london',0.30),('birmingham',0.10),('manchester',0.12),
              ('leeds',0.08),('glasgow',0.10),('cardiff',0.05),
              ('belfast',0.04)) t(loc, pop_weight) ON f.location = t.loc
WHERE timestamp_utc >= current_date
GROUP BY 1 ORDER BY 1;
import polars as pl

df = pl.read_parquet("data/silver/open_meteo/forecast_demand/**/*.parquet")
# Daily HDD totals per city (hourly hdd_k summed to daily degree-hours)
daily_hdd = (
    df.with_columns(pl.col("timestamp_utc").dt.date().alias("day"))
      .group_by(["day", "location"])
      .agg(pl.col("hdd_k").sum().alias("hdd_kh"))
      .pivot(index="day", on="location", values="hdd_kh")
      .sort("day")
)
print(daily_hdd.tail(7))
Caveats

Things to know before using this.

These are specific to forecast_demand. Also read the Open-Meteo-wide caveats on the vendor page.

01
No forecast_run_at — vintages overwrite silently

Dedup keeps the last silver row per (timestamp_utc, location), overwriting prior forecast vintages on re-ingest. Lead-time / forecast-skill backtests must pair against the historical archive at matching lead times. vault Known Issues · silver/openmeteo/historical.py L217

02
DISCREPANCY · vault schema vs silver
Silver column names are unit-suffixed (F15-B)

Silver emits temperature_2m_c, wind_speed_10m_mps (km/h→m/s ÷3.6), surface_pressure_hpa, hdd_k, etc. — the vault Silver Schema tables list the pre-rename names and are stale. Query the parquet with the suffixed forms. silver/openmeteo/historical.py L101-134 · _output_columns L255-271

03
Two-host design — forecast vs archive

This dataset uses api.open-meteo.com/v1/forecast. The archive sibling uses archive-api.open-meteo.com/v1/archive. past_days ≤ 92 is not used by the connector; for past data use historical_demand. connectors/openmeteo/client.py L90-94

04
No NWP-model pinning by default

Open-Meteo's /forecast router selects an NWP model per request (ECMWF / GFS / ICON / UKMO). The selection can change silently between calls; pin via &models=<id> if reproducibility matters. vault forecast_demand.md Known Issues "Unpinned model"

05
Three vendor-name forms coexist by convention

Vault folder is open-meteo (kebab); the Python package is openmeteo (no separator); the connector / registry key is open_meteo (snake) and is the bronze + silver path prefix. vault forecast_demand.md Known Issues "Naming inconsistency"

06
Bronze path uses DOUBLE underscore separator

Bronze partitions are bronze/open_meteo/forecast_demand__<site>/<year>/… — a double underscore between dataset key and site. Silver BRONZE_DATASET_PREFIX = "forecast_demand" strips it on read; custom bronze-readers must match the __ form or list zero files. connectors/openmeteo/client.py L112 · silver/openmeteo/historical.py L142
