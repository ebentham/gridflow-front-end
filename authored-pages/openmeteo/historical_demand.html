Overview

What this dataset is.

historical_demand holds the hourly ERA5-reanalysis weather that drives GB electricity demand — temperature, wind, humidity, pressure, precipitation and snow plus derived degree-hour features — at seven population-weighted GB cities, reaching back to 1940. It is the archive half of Gridflow's demand-weather pair: where forecast_demand looks forward off the NWP router, this feed is the stable, multi-decade reanalysis used for calibration. Each row is one hour at one GB site.

It is sourced from Open-Meteo's ERA5 archive endpoint at archive-api.open-meteo.com/v1/archive, queried once per city over a date range, validated against DemandWeather (schemas/weather.py), and written to silver via HistoricalDemandWeather. The payload is JSON columnar — parallel hourly arrays the transformer zips into one row per timestamp before the F15-B unit-suffix rename.

Multi-decade demand weather is the backtest backbone of the GB load model — ERA5 supplies the long HDD/CDD history that calibrates weather-adjusted load regressions, validated downstream against realised demand from Elexon indo. The forward sibling forecast_demand supplies the day-ahead nowcast. Verified against vendor docs: 2026-05-09 · Open-Meteo /archive.

-- Multi-year monthly HDD totals per city (multi-decade backtest backbone)
SELECT date_trunc('month', timestamp_utc) AS month,
       location,
       sum(hdd_k) AS hdd_kh,
       sum(cdd_k) AS cdd_kh
FROM read_parquet('data/silver/open_meteo/historical_demand/**/*.parquet')
WHERE timestamp_utc >= TIMESTAMP '2015-01-01'
GROUP BY 1, 2
ORDER BY 1, 2;
import polars as pl

df = pl.read_parquet("data/silver/open_meteo/historical_demand/**/*.parquet")
# Population-weighted GB-aggregate daily mean temperature (decade-scale)
gb_temp = (
    df.with_columns(pl.col("timestamp_utc").dt.date().alias("day"))
      .group_by(["day", "location"])
      .agg(pl.col("temperature_2m_c").mean())
      .pivot(index="day", on="location", values="temperature_2m_c")
      .with_columns(
          (pl.col("london")*0.30 + pl.col("birmingham")*0.10
         + pl.col("manchester")*0.12 + pl.col("leeds")*0.08
         + pl.col("glasgow")*0.10 + pl.col("cardiff")*0.05
         + pl.col("belfast")*0.04).alias("gb_pop_temp_c")
      )
      .sort("day")
)
print(gb_temp.tail(7))
Caveats

Things to know before using this.

These are specific to historical_demand. Also read the Open-Meteo-wide caveats on the vendor page.

01
ERA5 publication lag is ~5 days

Calls for end_date within the last 5 days may return 200 with null trailing values — ERA5 reanalysis cadence is non-real-time by design. Plan backfills accordingly; for very recent past use forecast_demand with past_days (≤92). vault Known Issues "ERA5 reanalysis lag"

02
DISCREPANCY · vault schema vs silver
Silver column names are unit-suffixed (F15-B)

Silver emits temperature_2m_c, wind_speed_10m_mps (km/h→m/s ÷3.6), surface_pressure_hpa, hdd_k, cdd_k, etc. — the vault Silver Schema tables list the pre-rename names and are stale. Query the parquet with the suffixed forms. silver/openmeteo/historical.py L101-134 · _output_columns L255-271

03
ERA5 grid-cell snapping

The latitude / longitude echoed in the response can differ slightly from the request — ERA5 snaps to the nearest grid cell (~0.25° resolution). Silver stores the response values, not the request values; don't dedup on lat/lon. vault historical_demand.md Known Issues "ERA5 grid-cell snapping"

04
Two-host design — archive lives at archive-api.open-meteo.com

The connector routes per dataset.startswith("historical") to ARCHIVE_BASE_URL, while the forecast sibling uses FORECAST_BASE_URL. The config/sources.yaml base_url is only valid for forecast; the archive path is selected by the dataset-prefix conditional. connectors/openmeteo/client.py L90-94 · endpoints.py L188-189

05
Three vendor-name forms coexist by convention

Vault folder is open-meteo (kebab); the Python package is openmeteo (no separator); the connector / registry key is open_meteo (snake) and is the bronze + silver path prefix. vault historical_demand.md Known Issues "Naming inconsistency"

06
Bronze path uses DOUBLE underscore separator

Bronze partitions are bronze/open_meteo/historical_demand__<site>/<year>/… — a double underscore between dataset key and site. Silver BRONZE_DATASET_PREFIX = "historical_demand" strips it on read; custom bronze-readers must match the __ form or list zero files. connectors/openmeteo/client.py L112 · silver/openmeteo/historical.py L142
