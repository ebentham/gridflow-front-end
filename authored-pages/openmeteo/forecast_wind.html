Overview

What this dataset is.

forecast_wind holds the hourly wind weather that drives GB wind generation — wind speed and direction at five heights (10 / 80 / 100 / 120 / 180 m), gusts, temperature, pressure, cloud and dew point — forecast 1–16 days ahead at twelve capacity-weighted GB sites (8 offshore clusters + 4 onshore). It is the forward-looking half of Gridflow's wind-weather pair, drawn from Open-Meteo's NWP router; unlike the ERA5 archive it carries the deep hub heights. Each row is one hour at one GB site.

It is sourced from Open-Meteo's forecast endpoint at api.open-meteo.com/v1/forecast, queried once per site, validated against WindWeather (schemas/weather.py), and written to silver via ForecastWindWeather. The payload is JSON columnar — parallel hourly arrays (hourly.time[], hourly.wind_speed_100m[], …) that the transformer zips into one row per timestamp, applying the km/h→m/s conversion and the F15-B unit-suffix rename.

Hub-height wind speed is the dominant driver of turbine power-curve output, and this feed is the day-ahead input to the wind-generation model: forecast speed here pairs with realised generation from Elexon fuelhh (the WIND field) and the NESO windfor public forecast to benchmark NWP skill, while the ERA5 sibling historical_wind supplies the multi-decade backtest. -- Day-ahead GB offshore mean 100m wind speed (capacity-weighted illustration)
SELECT date_trunc('hour', timestamp_utc) AS hour,
       avg(wind_speed_100m_mps) FILTER (WHERE location IN
         ('dogger_bank','hornsea','east_anglia','triton_knoll',
          'walney','gwynt_y_mor','beatrice','seagreen'))
         AS offshore_avg_100m_mps,
       avg(wind_speed_100m_mps) FILTER (WHERE location IN
         ('highland_central','borders_crystalrig','whitelee','pen_y_cymoedd'))
         AS onshore_avg_100m_mps
FROM read_parquet('data/silver/open_meteo/forecast_wind/**/*.parquet')
WHERE timestamp_utc >= current_date
GROUP BY 1 ORDER BY 1;
import polars as pl

df = pl.read_parquet("data/silver/open_meteo/forecast_wind/**/*.parquet")
# Power-curve-ready hub-height feature: prefer 100m, fall back to 10m x shear
power_input = (
    df.with_columns(
        pl.coalesce([
            pl.col("wind_speed_100m_mps"),
            pl.col("wind_speed_10m_mps") * 1.4,  # rough shear bump
        ]).alias("hub_height_mps"),
        # Density-corrected effective wind speed (power-curve work)
        (pl.col("hub_height_mps") * (pl.col("air_density_kg_m3") / 1.225) ** (1/3))
        .alias("rho_corrected_mps"),
    )
    .select(["timestamp_utc","location","hub_height_mps","rho_corrected_mps"])
)
print(power_input.tail(24))
Caveats

Things to know before using this.

These are specific to forecast_wind. Also read the Open-Meteo-wide caveats on the vendor page.

01
No forecast_run_at — vintages overwrite silently

Dedup keeps the last silver row per (timestamp_utc, location), overwriting prior forecast vintages on re-ingest. Lead-time backtests must pair against historical_wind at matching lead times. vault Known Issues · silver/openmeteo/historical.py L217

02
DISCREPANCY · vault schema vs silver
Silver column names are unit-suffixed (F15-B)

Silver emits wind_speed_10m_mps, wind_speed_100m_mps…180m_mps, wind_direction_*_deg, wind_gusts_10m_mps with km/h→m/s ÷3.6 — the vault Silver Schema tables list the pre-rename names and are stale. Query the parquet with the suffixed forms. silver/openmeteo/historical.py L101-134 · _output_columns L255-271

03
Hub-height nulls are model-driven, not silver bugs

wind_speed_80/120/180m_mps and matching directions go null when the NWP router selects a model that doesn't carry those heights (e.g. some GFS configs). Silver fabricates no values; WindWeather is permissive. Pin models=ukmo_global for stable hub-height coverage. vault forecast_wind.md Known Issues "Hub-height nulls"

04
No NWP-model pinning by default

Open-Meteo's /forecast router selects an NWP model per request (UKMO / ECMWF / GFS / ICON). Hub-height availability shifts with model selection; pin via &models=<id> if reproducibility matters. vault forecast_wind.md Known Issues "Unpinned NWP model"

05
Approximate site centroids (capacity-weighted, not NRO)

The 12 wind locations are capacity-weighted approximations per ADR-020, not per-turbine NRO coordinates. Sub-cluster resolution requires reading the ADR-020 trade-offs (vendor licensing + ergonomics). docs/DECISION_LOG/ADR-020-openmeteo-location-approximation.md

06
Bronze path uses DOUBLE underscore separator

Bronze partitions are bronze/open_meteo/forecast_wind__<site>/<year>/… — a double underscore between dataset key and site. Silver BRONZE_DATASET_PREFIX = "forecast_wind" strips it on read; custom bronze-readers must match the __ form or list zero files. connectors/openmeteo/client.py L112 · silver/openmeteo/historical.py L142
