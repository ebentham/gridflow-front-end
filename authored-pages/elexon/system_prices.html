Overview

What this dataset is.

system_prices holds the System Buy Price (SBP) and System Sell Price (SSP) — the cash-out prices used by the GB electricity Balancing Mechanism to settle imbalance. SBP is what generators pay (or are paid) for being short relative to their physical notifications; SSP is what suppliers pay for being long.

It is sourced from the BMRS Insights API at /balancing/settlement/system-prices/{settlementDate}, one date at a time. The raw JSON lands in bronze, is validated against the Pydantic schema, and written to the silver parquet partition via SystemPriceTransformer.

Together with the Net Imbalance Volume, this is the canonical signal for short-term GB power-market value and is the basis of the imbalance price in BSC settlement. Verified against vendor docs on 2026-05-09.

Schema

Silver layer columns.

Defined in gridflow/schemas/elexon.py · ElexonSystemPrice. The silver table is partitioned by settlement_date. Point-in-time field: run_type (precedence II<SF<R1<R2<R3<RF<DF).

Column Type Nullable Description
settlement_date PK date — Settlement date (BST/GMT calendar). Partition key.
settlement_period PK int — 1..50 (DST: 46 spring, 50 autumn).
timestamp_utc datetime[UTC] — Derived from (settlement_date, settlement_period) via utils/time.settlement_period_to_utc.
system_sell_price float — GBP/MWh. Range constraint: ge=-500, le=10000.
system_buy_price float — GBP/MWh. Typically equal to SSP under single-price reform (post-Nov 2018).
net_imbalance_volume float — MWh. Negative = system was short (needed more generation); positive = long.
run_type str null II / SF / R1 / R2 / R3 / RF / DF — BSC settlement run precedence. This endpoint does not surface the field, so live silver has run_type = None. See Settlement runs for what each code means.
price_derivation_code str null How SBP/SSP was derived for the period. Observed: N (normal), P (provisional). No regex constraint — vendor-managed list.
data_provider str — Default "elexon".
ingested_at datetime[UTC] null Time ingested into bronze. Useful for pipeline debugging.

Settlement runs

7 reconciliation passes.

BSC settlement runs each (settlement_date, settlement_period) multiple times as data is reconciled. Same key, different prices and NIV — the highest-rank run wins. This endpoint exposes the latest run's values but does not tag which run produced them; run_type stays null for silver derived from /system-prices/{date}.

II— Initial Indicative · within hours
SF— Settlement Final · settlement day
R1— Reconciliation 1 · ~10 days
R2— Reconciliation 2 · ~1 month
R3— Reconciliation 3 · ~4 months
RF— Reconciliation Final · ~14 months
DF— Post-Final Dispute · case-by-case

    "data/silver/elexon/system_prices/**/*.parquet",
)
# daily mean SBP and a min/max spread for volatility
daily = df.group_by("settlement_date").agg(
    pl.col("system_buy_price").mean().alias("sbp_mean"),
    (pl.col("system_buy_price").max() -
     pl.col("system_buy_price").min()).alias("sbp_range"),
)
print(daily.sort("settlement_date").tail())
Caveats

Things to know before using this.

These are specific to system_prices. Also read the Elexon-wide caveats on the vendor page.

01
Same key, different prices, over time

Same (settlement_date, settlement_period) reappears with different prices as BSC reconciliation progresses (II → SF → R1 → R2 → R3 → RF → DF). The transformer keeps the highest-rank run only. For point-in-time queries — "what did the trader see at 09:00 the next day?" — write a custom dedup or filter on the bronze layer's ingested_at.

02
SBP equals SSP under single-price reform

Since November 2018, GB cash-out uses a single price — SBP and SSP are typically equal in any given period. The two columns are kept for backward compatibility and edge cases where they diverge (e.g., emergency instructions). Treat system_buy_price as "the" imbalance price for most analysis.

03
price_derivation_code ≠ run_type

Earlier schema versions conflated priceDerivationCode with the BSC run type and applied a regex (^(II|SF|R[1-3]|RF|DF)$) that rejected the live values N and P. Resolved in V2 (V2-FIX-04, 2026-05-09): price_derivation_code is now a dedicated column with no regex; run_type is Optional[str]. Live silver from this endpoint leaves run_type null.

04
Settlement period range is 1..50

On the autumn clock change, the day has 50 settlement periods (25 hours). On spring forward, it has 46 (23 hours). DST handling required when bucketing by day or hour — the silver layer normalises all timestamps to UTC, but settlement_period can reach 50 on DST days.

05
NIV sign convention

Negative net_imbalance_volume means the system was short — actual generation undershot demand, the System Operator had to buy power, SBP went up. Positive NIV means long — generation overshot, the SO had to sell power, SSP went down (or negative in oversupply). Sign convention is GB-specific; do not assume it matches other markets.
