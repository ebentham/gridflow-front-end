Overview

What this dataset is.

boal holds Bid-Offer Acceptance Levels Flagged (BOALF) — every accepted bid or offer instruction issued by the National Electricity System Operator (NESO) to a Balancing Mechanism Unit. Each acceptance carries level-from / level-to MW values that fully describe how the unit's output was redispatched.

It is sourced from the BMRS Insights API at /datasets/BOALF, fetched by time window. The raw JSON lands in bronze and is processed via BOALTransformer into the silver parquet layer.

BOALF is the row-level audit trail behind headline BSC settlement numbers and is a primary feature for any BM-unit dispatch model. Data arrives near real-time (~10 min lag). Verified against vendor docs on 2026-05-08.

Settlement runs

7 reconciliation passes.

BOALF acceptances feed into BSC settlement, which is reconciled over 7 run types. The acceptance itself is real-time; its financial impact is settled across these runs.

II— Initial Indicative · within hours
SF— Settlement Final · settlement day
R1— Reconciliation 1 · ~10 days
R2— Reconciliation 2 · ~1 month
R3— Reconciliation 3 · ~4 months
RF— Reconciliation Final · ~14 months
DF— Post-Final Dispute · case-by-case

Caveats

Things to know before using this.

These are specific to boal. Also read the Elexon-wide caveats on the vendor page.

01
BM Unit IDs: preserve raw casing

Do NOT normalise BM Unit IDs — T_DRAXX-1 is different from t_draxx-1. Casing is part of the identifier and must be preserved for joins against bmunits_reference and other per-unit datasets.

02
An acceptance can span multiple settlement periods

settlementPeriodFrom to settlementPeriodTo can differ. Silver maps settlementPeriodFrom → settlement_period, losing the span info. Long-acceptance records may be deduped against shorter overlapping ones. For span-aware analysis, reconstruct from bronze.

03
Path renamed: BOAL → BOALF

The endpoint was renamed from /datasets/BOAL to /datasets/BOALF by Elexon. The vault slug stays boal for backward compatibility. The old bod and original boal endpoints are in EXCLUDED_ENDPOINTS.

04
Acceptance number is non-unique across (date, BM unit)

A given acceptance_number is not globally unique — it is an integer counter per day per unit. Proper deduplication requires the full combination of date, BM unit, acceptance number, and time fields.
